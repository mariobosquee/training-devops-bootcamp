CONFIGURACION

Configuracion inicial

Lo primero que debes hacer tras instalar Git es configurar tu identidad. Estos datos apareceran en cada commit que hagas.

bash
# Configurar nombre y email (global para todos los repos) git config --global user.name "Tu Nombre" git config --global user.email "tu@email.com" # Definir rama principal como "main" git config --global init.defaultBranch main # Activar colores en la terminal git config --global color.ui auto # Ver toda la configuracion actual git config --list
REPOSITORIO

Crear / clonar repositorio

Dos formas de empezar: crear un repo nuevo desde cero o clonar uno existente desde un remoto.

Crear un repositorio nuevo

bash
# Inicializar un repo en la carpeta actual git init # Inicializar un repo en una carpeta nueva git init mi-proyecto

Clonar un repositorio existente

bash
# Clonar por HTTPS git clone https://github.com/usuario/repo.git # Clonar por SSH git clone git@github.com:usuario/repo.git # Clonar en una carpeta con nombre personalizado git clone https://github.com/usuario/repo.git mi-carpeta
FLUJO BASICO

Flujo basico

Los comandos que usaras el 90% del tiempo. Domina estos y tendras el dia a dia cubierto.

Ver estado y cambios

bash
# Ver estado del working directory y staging area git status # Ver cambios no preparados (working dir vs staging) git diff # Ver cambios preparados (staging vs ultimo commit) git diff --staged

Preparar y confirmar cambios

bash
# Agregar un archivo al staging area git add archivo.txt # Agregar todos los cambios git add . # Crear un commit con mensaje git commit -m "feat: descripcion del cambio" # Add + commit en un solo paso (solo archivos tracked) git commit -am "fix: correccion rapida"

Sincronizar con el remoto

bash
# Subir commits al remoto git push # Bajar cambios del remoto y fusionar git pull # Subir rama nueva por primera vez git push -u origin nombre-rama

Historial

bash
# Ver historial de commits git log # Historial compacto en una linea git log --oneline # Historial con grafico de ramas git log --oneline --graph --all
RAMAS

Ramas

Las ramas te permiten trabajar en funcionalidades aisladas sin afectar la rama principal.

Crear y cambiar de rama

bash
# Ver todas las ramas (local) git branch # Ver ramas locales y remotas git branch -a # Crear una rama nueva git branch feature/login # Cambiar a otra rama git switch feature/login # Crear y cambiar en un solo paso git switch -c feature/login

Fusionar ramas

bash
# Primero, ir a la rama destino git switch main # Fusionar la rama feature en main git merge feature/login # Eliminar rama ya fusionada git branch -d feature/login
CONFLICTOS

Resolver conflictos

Cuando dos ramas modifican la misma linea, Git no puede decidir cual conservar. Tendras que resolverlo manualmente.

  • 1 Ejecutar git merge o git pull — Git avisa del conflicto
  • 2 Abrir el archivo conflictivo — buscar los marcadores <<<<<<<, ======= y >>>>>>>
  • 3 Editar el archivo y quedarte con el codigo correcto (eliminar los marcadores)
  • 4 Marcar como resuelto con git add archivo.txt
  • 5 Completar con git commit (Git crea el mensaje de merge automaticamente)

Ejemplo de marcadores de conflicto

archivo.txt
<<<<<<< HEAD console.log("version main"); ======= console.log("version feature"); >>>>>>> feature/login
bash
# Tras editar y resolver el conflicto: git add archivo.txt git commit
GITHUB

GitHub (trabajo remoto)

Comandos para gestionar la conexion con repositorios remotos y el flujo de Pull Requests.

Gestionar remotos

bash
# Ver remotos configurados git remote -v # Agregar un remoto git remote add origin https://github.com/usuario/repo.git # Cambiar la URL de un remoto git remote set-url origin https://github.com/usuario/nuevo-repo.git # Descargar info del remoto sin fusionar git fetch origin

Flujo de Pull Request

  • 1 Crear rama: git switch -c feature/nueva-funcionalidad
  • 2 Hacer commits en la rama
  • 3 Subir la rama: git push -u origin feature/nueva-funcionalidad
  • 4 Abrir Pull Request en GitHub (boton "Compare & pull request")
  • 5 Revisores aprueban o piden cambios
  • 6 Merge del PR en GitHub y eliminar la rama
DESHACER

Deshacer cambios

Todos cometemos errores. Git te permite retroceder en diferentes niveles segun donde esten los cambios.

Descartar cambios en working directory

bash
# Restaurar un archivo al ultimo commit git restore archivo.txt # Restaurar todos los archivos git restore .

Quitar del staging area (sin perder cambios)

bash
# Quitar un archivo del staging git restore --staged archivo.txt # Quitar todo del staging git restore --staged .

Deshacer commits

bash
# Deshacer ultimo commit (mantiene cambios en staging) git reset --soft HEAD~1 # Deshacer ultimo commit (mantiene cambios en working dir) git reset HEAD~1 # Deshacer ultimo commit (ELIMINA los cambios — cuidado!) git reset --hard HEAD~1
WORKFLOW

Flujo tipico

El workflow completo que seguiras en el dia a dia del bootcamp y en proyectos reales.

  • 1 git switch -c feature/mi-tarea — crear rama para tu tarea
  • 2 Editar archivos y hacer cambios
  • 3 git add . — preparar todos los cambios
  • 4 git commit -m "feat: descripcion" — confirmar los cambios
  • 5 git push -u origin feature/mi-tarea — subir la rama al remoto
  • 6 Abrir Pull Request en GitHub
  • 7 Code review y aprobacion del equipo
  • 8 Merge del PR a main
