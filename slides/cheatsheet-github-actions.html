GitHub Actions

Referencia rapida

1. Estructura de un workflow

Todo workflow vive en .github/workflows/nombre.yml y sigue esta estructura basica:

yaml
# Nombre visible en la pestana Actions name: CI Pipeline # Cuando se ejecuta on: push: branches: [main] # Trabajos a ejecutar jobs: build: runs-on: ubuntu-latest steps: - name: Checkout codigo uses: actions/checkout@v4 - name: Instalar dependencias run: npm install - name: Ejecutar tests run: npm test
  • name: — Nombre descriptivo del workflow.
  • on: — Evento(s) que lo disparan.
  • jobs: — Uno o mas trabajos. Cada job corre en su propia VM.
  • runs-on: — Sistema operativo (ubuntu-latest, windows-latest, macos-latest).
  • steps: — Lista ordenada de pasos dentro de un job.
  • uses: — Usa una action del marketplace.
  • run: — Ejecuta un comando de terminal.

2. Triggers

Los eventos que disparan un workflow:

yaml
# Push a ramas especificas on: push: branches: [main, develop] paths: - 'src/**' # Pull request on: pull_request: branches: [main] types: [opened, synchronize] # Ejecucion programada (cron) on: schedule: - cron: '0 6 * * 1' # Cada lunes a las 6:00 UTC # Ejecucion manual desde la UI on: workflow_dispatch: inputs: environment: description: 'Entorno destino' required: true default: 'staging' # Combinacion de triggers on: push: branches: [main] pull_request: branches: [main] workflow_dispatch:

3. Acciones comunes

Las actions mas utilizadas del marketplace:

yaml
# Clonar el repositorio - uses: actions/checkout@v4 # Configurar Node.js - uses: actions/setup-node@v4 with: node-version: '20' cache: 'npm' # Configurar Python - uses: actions/setup-python@v5 with: python-version: '3.12' cache: 'pip' # Cache generico de dependencias - uses: actions/cache@v4 with: path: ~/.cache/pip key: ${{ runner.os }}-pip-${{ hashFiles('**/requirements.txt') }} restore-keys: | ${{ runner.os }}-pip- # Build y push de imagen Docker a un registry - uses: docker/build-push-action@v5 with: context: . push: true tags: ghcr.io/${{ github.repository }}:latest

4. Secrets y variables

Gestionar credenciales y configuracion de forma segura:

yaml
# Variables de entorno a nivel de job jobs: deploy: runs-on: ubuntu-latest env: NODE_ENV: production API_URL: ${{ vars.API_URL }} steps: # Token automatico de GitHub (no requiere configuracion) - name: Login a GHCR run: echo ${{ secrets.GITHUB_TOKEN }} | docker login ghcr.io -u ${{ github.actor }} --password-stdin # Secret personalizado - name: Deploy run: ./deploy.sh env: AWS_ACCESS_KEY: ${{ secrets.AWS_ACCESS_KEY }} AWS_SECRET_KEY: ${{ secrets.AWS_SECRET_KEY }} # Variable de repositorio (no sensible) - name: Configurar entorno run: echo "Desplegando en ${{ vars.DEPLOY_ENV }}"
Donde configurar: Repositorio > Settings > Secrets and variables > Actions. Los secrets se enmascaran en los logs. Las variables (vars.) son visibles y se usan para configuracion no sensible.

5. Expresiones utiles

Condicionales, dependencias entre jobs y matrices:

yaml
# Condicional: ejecutar solo en main - name: Deploy if: github.ref == 'refs/heads/main' run: ./deploy.sh # Condicional: ejecutar solo si el job anterior tuvo exito - name: Notificar if: success() run: echo "Todo OK" # Condicional: ejecutar siempre (incluso si falla algo) - name: Cleanup if: always() run: docker system prune -f # Condicional: ejecutar solo si fallo - name: Alerta de fallo if: failure() run: curl -X POST ${{ vars.SLACK_WEBHOOK }} # Dependencia entre jobs (needs) jobs: test: runs-on: ubuntu-latest steps: - run: npm test deploy: needs: test # Espera a que "test" termine con exito runs-on: ubuntu-latest steps: - run: ./deploy.sh notify: needs: [test, deploy] # Espera a ambos jobs runs-on: ubuntu-latest steps: - run: echo "Pipeline completo" # Strategy matrix: ejecutar en multiples configuraciones jobs: test: runs-on: ubuntu-latest strategy: matrix: node-version: [18, 20, 22] os: [ubuntu-latest, windows-latest] steps: - uses: actions/setup-node@v4 with: node-version: ${{ matrix.node-version }} - run: npm test

6. Comandos utiles

Diferentes formas de ejecutar comandos en un step:

yaml
# Comando simple - run: npm install # Comando con nombre descriptivo - name: Instalar dependencias run: npm ci # Comando multilinea - name: Build y test run: | npm ci npm run build npm test # Cambiar directorio de trabajo - name: Tests del backend run: npm test working-directory: ./backend # Usar otra shell - name: Script en Python run: | import json print(json.dumps({"status": "ok"})) shell: python # Guardar output para otros steps - name: Obtener version id: version run: echo "tag=v1.0.0" >> $GITHUB_OUTPUT - name: Usar output anterior run: echo "Version es ${{ steps.version.outputs.tag }}"

7. Flujo tipico

Un pipeline CI/CD completo suele seguir este orden:

  1. Checkout — Clonar el codigo del repositorio.
  2. Setup — Instalar el runtime necesario (Node, Python, Go...).
  3. Install — Instalar dependencias del proyecto.
  4. Lint — Verificar estilo y calidad del codigo.
  5. Test — Ejecutar la suite de tests automatizados.
  6. Build — Compilar / construir la imagen Docker.
  7. Push — Subir la imagen al registry (GHCR, Docker Hub...).
  8. Deploy — Desplegar al entorno destino (staging, produccion).
  9. Notify — Enviar notificacion al equipo (Slack, email...).

8. Ejemplo minimo: CI basico

Un workflow completo listo para copiar y pegar:

yaml
name: CI Basico on: push: branches: [main] pull_request: branches: [main] jobs: ci: runs-on: ubuntu-latest steps: - name: Checkout codigo uses: actions/checkout@v4 - name: Setup Node.js uses: actions/setup-node@v4 with: node-version: '20' cache: 'npm' - name: Instalar dependencias run: npm ci - name: Lint run: npm run lint - name: Tests run: npm test - name: Build run: npm run build - name: Build imagen Docker if: github.ref == 'refs/heads/main' run: | docker build -t ghcr.io/${{ github.repository }}:${{ github.sha }} . echo ${{ secrets.GITHUB_TOKEN }} | docker login ghcr.io -u ${{ github.actor }} --password-stdin docker push ghcr.io/${{ github.repository }}:${{ github.sha }}
