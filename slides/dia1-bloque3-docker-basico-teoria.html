Docker Basico

Contenedores, imagenes, Dockerfile, volumenes, redes y Docker Compose

1. El problema que resuelve Docker

"En mi ordenador funciona."

El problema es que tu ordenador tiene unas versiones de software, unas configuraciones y unas librerias que no son las mismas que tiene el ordenador de tu companero, ni el servidor donde se despliega la aplicacion.

Docker resuelve esto: empaqueta tu aplicacion junto con todo lo que necesita para funcionar, de forma que se ejecuta exactamente igual en cualquier sitio.

2. Que es un contenedor

La analogia del contenedor de transporte maritimo

Piensa en un contenedor de un barco de carga:

  • Tiene un tamano estandar que encaja en cualquier barco, camion o tren.
  • Dentro puedes meter lo que quieras: fruta, muebles, electronica.
  • No importa que haya dentro: el contenedor se transporta siempre igual.
  • El contenido esta aislado y protegido del exterior.

Un contenedor de software funciona exactamente igual:

  • Tiene un formato estandar que funciona en cualquier ordenador con Docker.
  • Dentro lleva tu aplicacion y todo lo que necesita (librerias, configuracion, runtime).
  • No importa si el ordenador tiene Windows, Mac o Linux: el contenedor funciona igual.
  • La aplicacion esta aislada del resto del sistema.
Contenedor
+--------------------------------------------------+ | CONTENEDOR DE TRANSPORTE | | +--------------------------------------------+ | | | Tu aplicacion | | | | + Node.js 18 | | | | + Librerias (express, sqlite...) | | | | + Configuracion | | | | + Codigo fuente | | | +--------------------------------------------+ | +--------------------------------------------------+ Funciona igual en: tu portatil, el de tu companero, un servidor en la nube, un Raspberry Pi...

3. Contenedores vs Maquinas Virtuales

Los contenedores se parecen a las maquinas virtuales, pero son mucho mas ligeros.

Un contenedor es como un apartamento en un edificio (comparten estructura), mientras que una VM es como una casa independiente (cada una con sus cimientos).

4. Arquitectura de Docker

Docker tiene tres componentes principales:

Arquitectura
+----------------+ +------------------+ +------------------+ | | ordenes | | descarga | | | Docker Client | -------> | Docker Daemon | -------> | Docker Registry | | (CLI) | | (dockerd) | | (Docker Hub) | | | | | | | | Tu escribes | | El motor que | | Almacen de | | los comandos | | ejecuta todo | | imagenes | +----------------+ +------------------+ +------------------+ | +-------+-------+ | | +-----------+ +-----------+ | Contenedor| | Contenedor| | A | | B | +-----------+ +-----------+
  • Docker Client (CLI): Es lo que usas tu. Cuando escribes docker run, estas usando el cliente.
  • Docker Daemon (dockerd): Es el motor que corre en segundo plano. Recibe las ordenes del cliente, descarga imagenes, crea contenedores, etc.
  • Docker Registry (Docker Hub): Es como un "app store" de imagenes Docker. Ahi estan las imagenes oficiales de Node.js, Nginx, Python, etc.

5. Imagen vs Contenedor

Imagen = La receta

  • Es un archivo de solo lectura que contiene todo lo necesario para ejecutar una aplicacion.
  • Se define mediante un Dockerfile.
  • Se puede compartir y reutilizar.
  • No cambia una vez creada.

Contenedor = El plato cocinado

  • Es una instancia en ejecucion de una imagen.
  • Puedes crear muchos contenedores a partir de la misma imagen.
  • Cada contenedor es independiente.
  • Puede tener su propio estado (archivos modificados, datos, etc.).
Imagen vs Contenedor
IMAGEN (receta) CONTENEDORES (platos) +------------------+ +-------------+ +-------------+ | node:18-alpine | ------> | Contenedor 1| | Contenedor 2| | + tu codigo | ------> | (puerto 3001)| | (puerto 3002)| | + dependencias | ------> | Contenedor 3| +------------------+ +-------------+
La imagen es la receta de una tortilla de patatas. El contenedor es la tortilla que cocinas siguiendo esa receta. Puedes hacer 10 tortillas con la misma receta, y cada una es independiente.

6. El Dockerfile

Un Dockerfile es un archivo de texto que contiene las instrucciones para crear una imagen Docker. Es literalmente la receta paso a paso.

Instrucciones basicas

FROM — La imagen base

Dockerfile
FROM node:18-alpine

Le dice a Docker: "empieza con esta imagen base". node:18-alpine es una imagen ligera que ya trae Node.js 18 instalado.

WORKDIR — El directorio de trabajo

Dockerfile
WORKDIR /app

Establece el directorio donde se ejecutaran los siguientes comandos. Si no existe, la crea.

COPY — Copiar archivos

Dockerfile
COPY package*.json ./ COPY src/ ./src/

Copia archivos desde tu ordenador hacia dentro de la imagen.

RUN — Ejecutar comandos durante la construccion

Dockerfile
RUN npm install

Ejecuta un comando mientras se construye la imagen. Se ejecuta una vez, al construir la imagen, no cada vez que arrancas un contenedor.

EXPOSE — Documentar puertos

Dockerfile
EXPOSE 3001

Documenta que la aplicacion usa el puerto 3001. Es informativo: no abre el puerto automaticamente.

CMD — El comando de arranque

Dockerfile
CMD ["node", "src/index.js"]

Define que comando se ejecuta cuando arrancas un contenedor. Solo puede haber un CMD por Dockerfile.

Ejemplo completo: Dockerfile para nuestra API

Dockerfile
FROM node:18-alpine WORKDIR /app COPY package*.json ./ RUN npm install COPY src/ ./src/ EXPOSE 3001 CMD ["node", "src/index.js"]

Lectura en lenguaje natural: "Empieza con Node 18 Alpine, trabaja en /app, copia los archivos de dependencias, instala las dependencias, copia el codigo fuente, documenta el puerto 3001 y cuando arranques ejecuta la aplicacion."

7. Comandos esenciales de Docker

docker build — Construir una imagen

bash
docker build -t bootcamp-api .

-t bootcamp-api le da un nombre (tag) a la imagen. . indica que el Dockerfile esta en el directorio actual.

docker run — Crear y arrancar un contenedor

bash
docker run -p 3001:3001 bootcamp-api

-p 3001:3001 mapea el puerto 3001 de tu ordenador al 3001 del contenedor. Formato: -p PUERTO_HOST:PUERTO_CONTENEDOR. Anade -d para ejecutar en segundo plano.

docker ps — Ver contenedores en ejecucion

bash
docker ps

Muestra los contenedores que estan corriendo. Anade -a para ver tambien los parados.

docker stop — Parar un contenedor

bash
docker stop NOMBRE_O_ID

docker logs — Ver los logs

bash
docker logs NOMBRE_O_ID

Muestra lo que la aplicacion ha escrito en la consola. Anade -f para seguir los logs en tiempo real.

docker exec — Ejecutar un comando dentro de un contenedor

bash
docker exec -it NOMBRE_O_ID sh

Abre una terminal dentro del contenedor. Util para explorar que hay dentro.

8. Docker Compose

Que es Docker Compose

Una aplicacion real suele tener varios servicios: una API, un frontend, una base de datos... Docker Compose permite definir y ejecutar varios contenedores a la vez usando un unico archivo de configuracion: docker-compose.yml.

Por que usar Docker Compose

  • Un solo comando para levantar toda la aplicacion: docker compose up
  • Configuracion como codigo: todo queda documentado en el archivo YAML.
  • Redes automaticas: los contenedores pueden comunicarse entre si por nombre.
  • Facil de compartir: cualquier companero puede levantar todo el entorno con un solo comando.

Estructura basica de docker-compose.yml

docker-compose.yml
services: api: build: ./backend ports: - "3001:3001" environment: - NODE_ENV=production frontend: build: ./frontend ports: - "3000:80" depends_on: - api
  • services: define los contenedores que queremos ejecutar.
  • build: le dice que construya la imagen usando el Dockerfile de esa carpeta.
  • ports: mapea puertos, igual que -p en docker run.
  • environment: define variables de entorno.
  • depends_on: indica que el frontend necesita que la API este levantada primero.

Comandos de Docker Compose

bash
docker compose up -d # Levantar todos los servicios en segundo plano docker compose ps # Ver el estado de los servicios docker compose logs # Ver logs de todos los servicios docker compose logs api # Ver logs solo de la API docker compose down # Parar y eliminar todos los servicios docker compose build # Reconstruir las imagenes
