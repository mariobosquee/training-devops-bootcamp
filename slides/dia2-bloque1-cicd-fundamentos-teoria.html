CI/CD Fundamentos

Integracion continua, entrega continua y pipelines automatizados

1. El problema: desplegar a mano

Imagina que cada vez que haces un cambio en tu aplicacion tienes que:

  1. Conectarte al servidor por SSH.
  2. Descargar el codigo nuevo manualmente.
  3. Instalar dependencias.
  4. Parar la aplicacion.
  5. Arrancarla de nuevo.
  6. Rezar para que funcione.

Esto es lo que se llama despliegue manual. Es lento, propenso a errores y depende de que alguien se acuerde de todos los pasos. Si un viernes por la tarde alguien se salta el paso 3, la aplicacion se rompe y nadie sabe por que.

2. La solucion: automatizar con CI/CD

CI/CD son las siglas de tres conceptos relacionados pero distintos:

ConceptoSiglasQue significa
Continuous IntegrationCICada vez que alguien sube codigo, se ejecutan pruebas automaticamente para detectar errores rapido.
Continuous DeliveryCDAdemas de las pruebas, el codigo se empaqueta y queda listo para desplegar con un solo clic.
Continuous DeploymentCDEl paso final: el codigo se despliega automaticamente en produccion sin intervencion humana.

Piensa en ello como una escalera:

Escalera CI/CD
CI --> "El codigo funciona?" Continuous Delivery --> "El codigo esta listo para produccion?" Continuous Deployment--> "El codigo ya esta en produccion."

En este bootcamp llegaremos hasta Continuous Delivery: tu codigo se construira y empaquetara automaticamente. El despliegue lo haremos en bloques posteriores.

3. Las etapas de un pipeline

Un pipeline es la secuencia de pasos automaticos que recorre tu codigo desde que lo subes hasta que esta desplegado. Las etapas tipicas son:

Pipeline
Code ---> Build ---> Test ---> Deploy | | | | | | | +-- Poner en produccion | | +-------------- Ejecutar pruebas automaticas | +-------------------------- Compilar / empaquetar +-------------------------------------- Subir cambios al repositorio

Cada etapa es como una puerta de control: si una falla, las siguientes no se ejecutan. Esto garantiza que solo codigo que funciona llegue a produccion.

4. Analogia: la cadena de montaje

CI/CD es como una cadena de montaje en una fabrica:

  • Code = la materia prima llega a la fabrica (tu codigo nuevo).
  • Build = las piezas se ensamblan (se construye la imagen Docker).
  • Test = control de calidad revisa cada pieza (las pruebas automaticas comprueban que todo funciona).
  • Deploy = el producto terminado sale al mercado (la aplicacion se despliega para los usuarios).
Sin cadena de montaje, un operario tendria que hacer todo a mano: lento, caro y con muchos defectos. Con la cadena, el proceso es rapido, repetible y fiable.

5. Antes vs Despues

Sin CI/CDCon CI/CD
Despliegue manual cada vezDespliegue automatico en cada push
Errores se descubren en produccionErrores se descubren en el pipeline
"En mi maquina funciona"Funciona en un entorno limpio y controlado
Desplegar da miedoDesplegar es rutinario
Un deploy cada semana o mesMultiples deploys al dia

6. Resumen

  • CI = integrar y probar codigo automaticamente.
  • CD (Delivery) = empaquetar y dejar listo para desplegar.
  • CD (Deployment) = desplegar automaticamente.
  • Un pipeline es la secuencia: Code → Build → Test → Deploy.
  • CI/CD elimina los errores humanos y acelera el ciclo de desarrollo.
