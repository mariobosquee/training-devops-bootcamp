Parte 1

Probar lint en local

1 Verificad que lint funciona en local:

cd frontend
npm install
npm run lint
# Debeis ver el resultado del linter (0 errores idealmente)

2 Anadid un job lint en paralelo con test:

  lint:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout codigo
        uses: actions/checkout@v4

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '18'

      - name: Instalar dependencias frontend
        run: npm install
        working-directory: ./frontend

      - name: Ejecutar linter
        run: npm run lint
        working-directory: ./frontend

Parte 1

Build depende de lint y test

3 Actualizad el job build para que dependa de ambos:

  build:
    needs: [lint, test]
    runs-on: ubuntu-latest
    steps:
      # ... (el resto del job build igual que antes)
Clave

needs: [lint, test] hace que lint y test se ejecuten en paralelo, y build solo empieza cuando ambos han pasado.

4 Push y verificar:

git add .github/workflows/ci.yml
git commit -m "ci: add lint job in parallel"
git push
Si algo falla
lint falla con errores
Corregid los errores en local con npm run lint -- --fix
No hay script lint
Anadid "lint": "eslint ." en package.json del frontend

Parte 2

Anadir npm audit

1 Probad en local:

cd backend
npm audit
# Muestra vulnerabilidades conocidas en las dependencias

2 Anadid un step de seguridad al job test:

      - name: Security audit
        run: npm audit --audit-level=moderate
        working-directory: ./backend
        continue-on-error: true
Por que continue-on-error?

continue-on-error: true permite que el pipeline siga aunque haya vulnerabilidades. En produccion podeis quitarlo para bloquear el deploy si hay vulnerabilidades criticas.

Parte 2

Verificar el scanning

3 Push y verificar:

git add .github/workflows/ci.yml
git commit -m "ci: add security audit step"

4 En Actions, abrid el job test y verificad:

Parte 3

Anadir cache de dependencias

1 Anotad el tiempo actual del job test en la pestana Actions.

2 Anadid cache en el job test (despues de checkout, antes de npm install):

      - name: Cache node modules
        uses: actions/cache@v4
        with:
          path: ./backend/node_modules
          key: ${{ runner.os }}-node-backend-${{ hashFiles('backend/package-lock.json') }}
          restore-keys: |
            ${{ runner.os }}-node-backend-

3 Haced lo mismo para el job lint (con path del frontend):

      - name: Cache node modules
        uses: actions/cache@v4
        with:
          path: ./frontend/node_modules
          key: ${{ runner.os }}-node-frontend-${{ hashFiles('frontend/package-lock.json') }}
          restore-keys: |
            ${{ runner.os }}-node-frontend-

Parte 3

Comparar tiempos

4 Push y ejecutar dos veces:

git add .github/workflows/ci.yml
git commit -m "ci: add npm cache"
git push

5 Primera ejecucion: la cache se crea (no hay mejora aun).

6 Haced un cambio menor y pushed de nuevo:

# Anadid un comentario al README o un archivo
git commit --allow-empty -m "ci: test cache speed"
git push

7 Comparad los tiempos: la segunda ejecucion deberia ser mas rapida en npm install.

Si algo falla
Cache miss siempre
Comprobad que la ruta de path: coincide con donde se instalan los modulos
No hay mejora visible
Normal si el proyecto es pequeno. La mejora es mayor con muchas dependencias

Parte 4

Anadir el badge

1 Obtener la URL del badge:

  • Id a la pestana Actions en GitHub
  • Click en el nombre del workflow (CI Pipeline)
  • Click en el boton "..." (tres puntos) arriba a la derecha
  • Seleccionad "Create status badge"
  • Copiad el Markdown

2 Anadid al principio de README.md:

![CI Pipeline](https://github.com/USUARIO/REPO/actions/workflows/ci.yml/badge.svg)

3 Push:

git add README.md
git commit -m "docs: add CI status badge"
git push

Verificad que el badge aparece en la pagina principal del repositorio.

Bonus

Bonus: Service container MongoDB

Si terminais antes de tiempo, anadid un servicio MongoDB al job de test:

  test:
    runs-on: ubuntu-latest
    services:
      mongodb:
        image: mongo:7
        ports:
          - 27017:27017
    steps:
      # ... vuestros steps de siempre
      - name: Ejecutar tests
        run: npm test
        working-directory: ./backend
        env:
          MONGODB_URI: mongodb://localhost:27017/bookstore_test
Service containers

GitHub Actions permite levantar servicios (como bases de datos) al lado de vuestro codigo. Funcionan como contenedores Docker que se crean y destruyen con cada ejecucion.
