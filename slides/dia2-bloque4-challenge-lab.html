Challenge 1

Matrix: Node 18 y 20

Objetivo: Ejecutar los tests con Node.js 18 y Node.js 20 en paralelo.

1 Anadid una strategy matrix al job test:

  test:
    runs-on: ubuntu-latest
    strategy:
      matrix:
        node-version: [18, 20]
    steps:
      - name: Checkout codigo
        uses: actions/checkout@v4

      - name: Setup Node.js ${{ matrix.node-version }}
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}

      - name: Instalar dependencias
        run: npm install
        working-directory: ./backend

      - name: Ejecutar tests
        run: npm test
        working-directory: ./backend
Resultado esperado

En Actions vereis dos ejecuciones paralelas del job test: una con Node 18 y otra con Node 20.

Challenge 2

Jobs lint y test en paralelo

Objetivo: Crear dos jobs separados que corran en paralelo, y que build dependa de ambos.

1 Estructura del workflow:

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '18'
      - run: npm install
        working-directory: ./frontend
      - run: npm run lint
        working-directory: ./frontend

  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '18'
      - run: npm install
        working-directory: ./backend
      - run: npm test
        working-directory: ./backend

  build:
    needs: [lint, test]
    runs-on: ubuntu-latest
    steps:
      # ... build steps
Resultado esperado

En el grafo de Actions, lint y test aparecen al mismo nivel, y build empieza despues de que ambos terminen.

Challenge 3

Deploy condicional

Objetivo: Crear un job deploy que solo se ejecute en pushes a main.

1 Anadid un job deploy con condicion if:

  deploy:
    needs: build
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - name: Checkout codigo
        uses: actions/checkout@v4

      - name: Simular deploy
        run: |
          echo "Deploying to production..."
          echo "Branch: ${{ github.ref }}"
          echo "Commit: ${{ github.sha }}"
          echo "Deploy completed!"

2 Para probar: cread una rama, pushed a ella, y verificad que deploy NO se ejecuta. Luego merged a main y verificad que SI se ejecuta.

Resultado esperado

El job deploy aparece en gris (skipped) cuando el push es a una rama que no es main.

Challenge 4

Crear y usar secrets

Objetivo: Crear un secret en GitHub y usarlo en el workflow.

1 Crear el secret en GitHub:

  • Settings → Secrets and variables → Actions
  • Click en "New repository secret"
  • Nombre: DEPLOY_ENV — Valor: production

2 Usad el secret en un step:

      - name: Usar secret
        run: |
          echo "Deploy environment: $DEPLOY_ENV"
          echo "El valor real esta oculto en los logs"
        env:
          DEPLOY_ENV: ${{ secrets.DEPLOY_ENV }}
Resultado esperado

En los logs, el valor del secret aparece como ***. GitHub lo oculta automaticamente para proteger la informacion sensible.

Challenge 5

PR con auto-comment

Objetivo: Crear un workflow que se ejecute en pull requests y deje un comentario automatico.

1 Crear .github/workflows/pr.yml:

name: PR Check

on:
  pull_request:
    branches: [main]

permissions:
  contents: read
  pull-requests: write

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout codigo
        uses: actions/checkout@v4

      - name: Ejecutar tests
        run: |
          cd backend
          npm install
          npm test

      - name: Comentar en el PR
        uses: actions/github-script@v7
        with:
          script: |
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner,
              repo: context.repo.repo,
              body: 'Tests passed! Ready for review.'
            })

2 Probad: cread una rama, pushed, abrid un PR, y verificad el comentario automatico.

Ayuda

Errores comunes

Problemas frecuentes
Error de YAML: "mapping values are not allowed"
Indentacion incorrecta. Usad 2 espacios, nunca tabs
Job skipped inesperadamente
Revisad la condicion if: y que needs: apunte al job correcto
Matrix no se ejecuta en paralelo
Comprobad que strategy: y matrix: estan bien indentados dentro del job
Secret no disponible
Los secrets no estan disponibles en forks. Usad vuestro propio repositorio
PR comment no aparece
Comprobad permissions: pull-requests: write en el workflow
"Resource not accessible by integration"
Falta el bloque permissions: al nivel raiz del workflow
