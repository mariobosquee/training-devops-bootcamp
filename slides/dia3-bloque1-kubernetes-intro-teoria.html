Kubernetes Intro

Referencia teorica para estudiantes

1. El problema: demasiados contenedores

En el Dia 1 dockerizamos nuestra app. En el Dia 2 automatizamos la construccion de imagenes con CI/CD. Ahora tenemos imagenes Docker listas para produccion, pero surge un problema nuevo:

  • Si tenemos 10 microservicios, cada uno con 3 replicas, son 30 contenedores.
  • Si un contenedor se cae a las 3 de la manana, quien lo reinicia?
  • Si hay mas trafico, quien lanza mas contenedores?
  • Si actualizamos una version, quien para los viejos y arranca los nuevos?

Hacer todo esto a mano con docker run es inviable. Necesitamos un orquestador de contenedores.

2. Que es Kubernetes

Kubernetes (K8s) es un sistema de orquestacion de contenedores creado por Google. Su trabajo es:

  • Desplegar contenedores automaticamente.
  • Escalar (mas o menos replicas segun la carga).
  • Recuperarse de fallos (self-healing).
  • Actualizar sin downtime (rolling updates).

Analogia para entenderlo:

Kubernetes es como el director de un hotel.

- Los pods son las habitaciones (cada una tiene un huesped = contenedor).
- El director asigna habitaciones a los huespedes.
- Si una habitacion tiene un problema, el director reubica al huesped en otra.
- Si llegan muchos huespedes, el director abre mas habitaciones.
- Si un huesped se va, la habitacion se libera automaticamente.

Tu no gestionas las habitaciones una a una. Le dices al director:
"Quiero 3 habitaciones con vista al mar" y el se encarga.

3. Arquitectura simplificada

Kubernetes tiene dos partes principales:

+-----------------------------------------------+
|              CONTROL PLANE (cerebro)           |
|                                                |
|  - API Server: recibe tus ordenes (kubectl)    |
|  - Scheduler: decide donde va cada pod         |
|  - Controller Manager: vigila que todo este OK |
|  - etcd: base de datos del estado del cluster  |
+-----------------------------------------------+
          |              |              |
   +------+------+ +----+-------+ +----+-------+
   | Worker Node | | Worker Node | | Worker Node |
   |  (musculo)  | |  (musculo)  | |  (musculo)  |
   |             | |             | |             |
   | [Pod] [Pod] | | [Pod] [Pod] | | [Pod]       |
   +-------------+ +-------------+ +-------------+
  • Control Plane (cerebro): toma decisiones. Tu le hablas a el.
  • Worker Nodes (musculos): ejecutan los contenedores. Hacen el trabajo real.

En nuestro laboratorio usaremos kind (Kubernetes IN Docker), que simula todo esto en tu portatil.

4. Conceptos clave

Solo necesitas conocer cinco conceptos para empezar:

ConceptoQue esAnalogia hotel
PodLa unidad minima de K8s. Contiene uno o mas contenedores.Una habitacion
DeploymentGestiona un grupo de pods. Controla cuantas replicas hay, las actualiza y las recupera si fallan.El plano de planta
ServiceUna direccion de red estable para acceder a los pods. Los pods van y vienen, pero el Service siempre apunta a los correctos.El telefono de recepcion
ConfigMapAlmacena configuracion (variables de entorno, archivos de config) separada del contenedor.El manual de instrucciones
NamespaceOrganiza los recursos del cluster en grupos logicos.Las plantas del hotel

5. kubectl: tu herramienta principal

kubectl es el CLI que usamos para hablar con Kubernetes. Todos los comandos siguen el mismo patron:

kubectl <verbo> <recurso> [nombre] [opciones]

Comandos esenciales:

# Ver recursos
kubectl get pods                    # Lista pods
kubectl get deployments             # Lista deployments
kubectl get services                # Lista services
kubectl get all                     # Lista todo

# Inspeccionar
kubectl describe pod <nombre>       # Detalle de un pod
kubectl logs <nombre>               # Ver logs de un contenedor

# Crear/modificar
kubectl apply -f archivo.yaml       # Aplica un manifiesto YAML
kubectl delete -f archivo.yaml      # Elimina lo definido en el YAML

# Depurar
kubectl exec -it <pod> -- bash      # Entrar al contenedor
kubectl port-forward pod/<pod> 8080:80  # Redirigir puerto local al pod

Resumen

Antes (Docker manual)Ahora (Kubernetes)
Tu decides donde corre cada contenedorK8s lo decide automaticamente
Si un contenedor se cae, lo reinicias tuK8s lo reinicia solo (self-healing)
Escalas a mano con docker runK8s escala con kubectl scale
Actualizas parando todoK8s hace rolling updates sin downtime
Todo en un solo servidorDistribuido en multiples nodos
