Persistencia, Deployment y Service para la base de datos

carlospalanca.es
04 / 20
Parte 1

Crear carpeta y PVC

1 Crear la carpeta de manifiestos:

mkdir -p k8s
cd campuseats

2 Crear k8s/mongo-pvc.yaml:

apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: mongo-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 1Gi
Que es un PVC

Un PersistentVolumeClaim pide almacenamiento al cluster. Los datos de MongoDB sobreviviran si el pod se reinicia.

05 / 20
Parte 1

Deployment y Service de MongoDB

3 Crear k8s/mongo-deployment.yaml:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: mongo
spec:
  replicas: 1
  selector:
    matchLabels:
      app: mongo
  template:
    metadata:
      labels:
        app: mongo
    spec:
      containers:
        - name: mongo
          image: mongo:7
          ports:
            - containerPort: 27017
          volumeMounts:
            - name: mongo-storage
              mountPath: /data/db
      volumes:
        - name: mongo-storage
          persistentVolumeClaim:
            claimName: mongo-pvc

4 Crear k8s/mongo-service.yaml:

apiVersion: v1
kind: Service
metadata:
  name: mongo-svc
spec:
  selector:
    app: mongo
  ports:
    - port: 27017
      targetPort: 27017

Parte 1

Aplicar y verificar MongoDB

5 Aplicar los manifiestos:

kubectl apply -f k8s/mongo-pvc.yaml
kubectl apply -f k8s/mongo-deployment.yaml
kubectl apply -f k8s/mongo-service.yaml

6 Verificar que todo esta corriendo:

kubectl get pods
# mongo-xxxx  STATUS: Running

kubectl get svc
# mongo-svc  ClusterIP  27017/TCP
Si algo falla
PVC en Pending
kind incluye un StorageClass por defecto. Esperad 30s
Pod en CrashLoopBackOff
Comprobad logs: kubectl logs mongo-xxxx
Error en YAML
Revisad la indentacion. YAML usa espacios, no tabuladores

Parte 2

Cargar imagen y crear Deployment

1 Cargar la imagen en el cluster de kind:

kind load docker-image campuseats-backend:v1 --name bootcamp
Por que kind load

kind ejecuta K8s dentro de Docker. Las imagenes locales no son visibles dentro del cluster. kind load las copia al nodo.

2 Crear k8s/backend-deployment.yaml:

apiVersion: apps/v1
kind: Deployment
metadata:
  name: backend
spec:
  replicas: 1
  selector:
    matchLabels:
      app: backend
  template:
    metadata:
      labels:
        app: backend
    spec:
      containers:
        - name: backend
          image: campuseats-backend:v1
          imagePullPolicy: Never
          ports:
            - containerPort: 3000
          env:
            - name: DATABASE_URI
              value: "mongodb://mongo-svc:27017/campuseats"

Parte 4

Crear y usar ConfigMap

1 Crear k8s/configmap.yaml:

apiVersion: v1
kind: ConfigMap
metadata:
  name: campuseats-config
data:
  DATABASE_URI: "mongodb://mongo-svc:27017/campuseats"
  NODE_ENV: "production"
  PORT: "3000"

2 Actualizar k8s/backend-deployment.yaml — reemplazad la seccion env por envFrom:

      containers:
        - name: backend
          image: campuseats-backend:v1
          imagePullPolicy: Never
          ports:
            - containerPort: 3000
          envFrom:
            - configMapRef:
                name: campuseats-config

3 Aplicar todo de golpe:

kubectl apply -f k8s/

4 Verificar que las variables estan inyectadas:

kubectl exec deploy/backend -- env | grep DATABASE
# Debeis ver: DATABASE_URI=mongodb://mongo-svc:27017/campuseats
Ventaja

Ahora podeis cambiar la configuracion sin reconstruir la imagen Docker. Solo editais el ConfigMap y reiniciais el pod.

Rolling update y rollback

Desplegar una nueva version sin downtime

carlospalanca.es
15 / 20
Parte 5

Construir v2 y desplegar

1 Haced un cambio visible en el codigo del backend (por ejemplo, cambiad el mensaje de health check o anadid un endpoint).

2 Construir la imagen v2:

docker build -t campuseats-backend:v2 ./backend

3 Cargar en kind:

kind load docker-image campuseats-backend:v2 --name bootcamp

4 Actualizar el deployment:

kubectl set image deployment/backend backend=campuseats-backend:v2

5 Observar el rolling update en directo:

# En una terminal:
kubectl get pods -w

# En otra terminal:
kubectl rollout status deployment/backend
# Debeis ver: deployment "backend" successfully rolled out

16 / 20
Parte 5

Verificar y hacer rollback

6 Verificar la nueva version:

kubectl port-forward svc/backend-svc 3000:3000
# Comprobad que el cambio esta visible
# Parad con Ctrl+C

7 Hacer rollback a v1:

kubectl rollout undo deployment/backend

8 Verificar que volvemos a v1:

kubectl rollout status deployment/backend
kubectl port-forward svc/backend-svc 3000:3000
# Comprobad que el cambio ha desaparecido — estamos en v1
Clave

Kubernetes mantiene un historial de versiones. Con rollout undo podeis volver a la version anterior en segundos, sin reconstruir nada.

Si algo falla
El pod nuevo no arranca
Comprobad kubectl describe pod backend-xxxx y kubectl logs backend-xxxx
No se ve el cambio
Verificad la imagen: kubectl describe deploy backend | grep Image
