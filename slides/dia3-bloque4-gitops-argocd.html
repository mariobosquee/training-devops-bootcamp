El problema

El problema

Despliegues manuales
  • Cada persona ejecuta `kubectl apply` con sus ficheros locales.
  • No hay registro claro de quien cambio que.
  • Si algo se rompe, no es facil volver atras.
  • Cada miembro del equipo puede tener versiones diferentes.

Necesitamos una fuente de verdad unica.

Que es GitOps

Que es GitOps

Git = la unica fuente de verdad
  • Todo lo que debe estar en el cluster se define en Git.
  • Para cambiar el cluster, haces un commit.
  • Una herramienta automatica detecta el cambio y actualiza el cluster.

No ejecutas comandos contra el cluster: editas Git.

Analogia: Google Doc

Analogia: Google Docs

GitOps es como Google Docs
  • Alguien edita el documento --> todos ven el cambio al instante.
  • No hace falta enviar ficheros por correo ni avisar a nadie.
  • El documento compartido es la fuente de verdad.

En GitOps:

  • Alguien hace commit + push --> el cluster se actualiza automaticamente.
  • El repositorio Git es la fuente de verdad.

Push vs Pull

Push vs Pull

Dos modelos para aplicar cambios

Modelo Push:

  • Una pipeline externa ejecuta `kubectl apply` tras cada commit.
  • El sistema "empuja" los cambios al cluster.

Modelo Pull:

  • Una herramienta dentro del cluster vigila Git.
  • Cuando detecta cambios, "tira" de ellos y los aplica.
  • Mas seguro: nadie externo necesita credenciales del cluster.

ArgoCD usa el modelo Pull.
