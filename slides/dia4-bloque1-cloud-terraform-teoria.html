Cloud y Terraform

Infraestructura como codigo

1. Que es Cloud Computing

En los tiempos antiguos, si querias montar una web tenias que comprar un servidor fisico, ponerlo en una sala con aire acondicionado, conectarle cables, instalar el sistema operativo y mantenerlo tu mismo. Si se rompia el disco duro a las 3 de la manana, problema tuyo.

Cloud Computing es alquilar ordenadores por internet en lugar de comprarlos. Alguien (Amazon, Microsoft, Google) tiene miles de servidores en centros de datos gigantes y tu los usas a demanda: pagas solo por lo que consumes, como la luz de tu casa.

2. IaaS, PaaS y SaaS

Hay tres formas de "consumir" cloud, y se entienden mejor con pizza:

ModeloQue te danAnalogia pizza
IaaSMaquinas virtuales, redes, discos. Tu instalas todo.Te dan el horno y los ingredientes. Tu haces la pizza.
PaaSEntorno listo para subir tu codigo. No tocas servidores.Te dan una pizza congelada. Tu solo la metes al horno.
SaaSAplicacion terminada. Solo la usas.Pizza a domicilio. Abres la caja y comes.

4. El problema: infraestructura manual

Hasta ahora, cuando alguien necesita infraestructura en cloud, hace esto:

  • Abre la consola web (portal de AWS/Azure/GCP).
  • Hace clic en "Crear maquina virtual".
  • Rellena 15 formularios.
  • Le da a "Crear".
  • Repite para la base de datos, el balanceador, la red...

Problemas de hacerlo asi:

  • Lento: crear 50 recursos a mano lleva horas.
  • Propenso a errores: un clic mal y la configuracion es distinta.
  • No reproducible: si quieres crear lo mismo en otro entorno, tienes que repetir todos los clics.
  • Sin historial: no sabes quien cambio que ni cuando.

Suena al mismo problema que teniamos antes de Git, verdad?

5. Infrastructure as Code (IaC)

La solucion es la misma idea que con el codigo: escribir la infraestructura en ficheros de texto y gestionarlos con control de versiones.

Infraestructura manual          Infrastructure as Code
------------------------------------------------------
Clicks en la consola web   -->  Ficheros .tf en un repo Git
"Creo que lo configure asi" --> Codigo versionado y revisable
Imposible de reproducir    -->  terraform apply y listo

6. Que es Terraform

Terraform es una herramienta de HashiCorp que permite crear, modificar y destruir infraestructura escribiendo codigo. Sus ventajas:

  • Multi-cloud: funciona con AWS, Azure, GCP, Docker, Kubernetes y cientos mas.
  • Declarativo: tu describes QUE quieres, no COMO crearlo.
  • Estado: Terraform recuerda que ha creado (fichero terraform.tfstate).
  • Plan: antes de hacer cambios, te ensena lo que VA a hacer.

7. HCL: el lenguaje de Terraform

Terraform usa un lenguaje propio llamado HCL (HashiCorp Configuration Language). Estos son los bloques basicos:

# 1. TERRAFORM BLOCK: que proveedores necesitamos
terraform {
  required_providers {
    local = {
      source  = "hashicorp/local"
      version = "2.4.1"
    }
  }
}

# 2. PROVIDER: configuracion del proveedor
provider "local" {}

# 3. RESOURCE: lo que queremos crear
resource "local_file" "saludo" {
  content  = var.mensaje        # <-- usa una variable
  filename = "saludo.txt"
}

# 4. VARIABLE: parametros de entrada
variable "mensaje" {
  type    = string
  default = "Hola desde Terraform"
}

# 5. OUTPUT: valores de salida (para ver o usar despues)
output "ruta_fichero" {
  value = local_file.saludo.filename
}

8. El flujo de trabajo de Terraform

Terraform tiene 4 comandos principales que siempre se ejecutan en este orden:

terraform init       Descarga los proveedores necesarios.
    |                (Solo la primera vez o al cambiar proveedores.)
    v
terraform plan       Muestra QUE va a hacer, sin tocar nada.
    |                (Vista previa de los cambios.)
    v
terraform apply      Ejecuta los cambios de verdad.
    |                (Crea/modifica/destruye recursos.)
    v
terraform destroy    Elimina TODO lo que Terraform creo.
                     (Limpieza total.)

Regla de oro: nunca hagas apply sin mirar el plan antes.
