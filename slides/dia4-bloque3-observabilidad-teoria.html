Observabilidad

Prometheus, Grafana y las Golden Signals

1. El problema: la app esta corriendo, pero esta bien?

Hasta ahora hemos conseguido mucho: tenemos nuestra app CampusEats dockerizada, un pipeline CI/CD que la construye automaticamente, Kubernetes que la despliega, ArgoCD que sincroniza desde Git y Terraform que gestiona infraestructura como codigo.

Pero hay una pregunta fundamental que no sabemos responder:

  • La app esta respondiendo rapido o va lenta?
  • Cuantas peticiones esta recibiendo por minuto?
  • Esta dando errores 500 a los usuarios?
  • La CPU o la memoria estan al limite?

Sin observabilidad, tu app es una caja negra. Funciona hasta que alguien te llama para decirte que no funciona. Con observabilidad, tu ves los problemas antes que los usuarios.

2. Los tres pilares de la observabilidad

La observabilidad se apoya en tres tipos de datos:

PilarQue esEjemplo
MetricasNumeros que cambian con el tiempoCPU al 75%, 200 peticiones/min, 2% errores
LogsMensajes de texto que genera la app"Error: conexion a base de datos rechazada"
TrazasEl camino que sigue una peticion a traves de varios serviciosUsuario -> API -> Base de datos -> Respuesta

En este bloque nos centramos en metricas, que es lo mas visual y lo que mas impacto tiene para empezar.

3. Las cuatro Golden Signals

Google definio cuatro senales clave para saber si un servicio esta sano:

  • Latencia: cuanto tarda en responder. Si normalmente tarda 50ms y ahora tarda 2 segundos, algo va mal.
  • Trafico: cuantas peticiones recibe. Sirve para saber si hay picos de uso o si nadie esta usando la app.
  • Errores: que porcentaje de respuestas son errores (codigos 500, timeouts). Si sube del 1%, hay que investigar.
  • Saturacion: cuanto de llenos estan los recursos (CPU, memoria, disco). Si la CPU esta al 95%, la app esta al limite.

Con estas cuatro senales puedes diagnosticar el 90% de los problemas.

4. Prometheus: el recolector de metricas

Prometheus es una herramienta que se conecta a tus aplicaciones y recoge metricas automaticamente. Funciona asi:

  • Tus apps exponen un endpoint /metrics con numeros (peticiones, errores, latencia, etc.).
  • Prometheus visita ese endpoint cada pocos segundos y guarda los datos.
  • Puedes consultar esos datos para saber que esta pasando.

Nuestra app CampusEats ya tiene instrumentacion con prom-client. Si visitas http://localhost:3333/metrics, veras las metricas que expone.

5. Grafana: los dashboards bonitos

Grafana es una herramienta de visualizacion. Se conecta a Prometheus y convierte los numeros en graficos, tablas y dashboards interactivos.

  • Puedes ver cuantas peticiones recibe CampusEats por segundo.
  • Puedes ver la latencia (P50, P95, P99) de cada endpoint.
  • Puedes ver cuantos platos hay en la base de datos.
  • Puedes ver CPU y memoria del servidor.

Grafana viene con dashboards pre-configurados para Kubernetes, asi que no hay que crear nada desde cero.

6. La analogia: el termometro y la pantalla

Prometheus = el termometro
  - Va midiendo la temperatura (metricas) de cada parte del sistema.
  - Guarda un historico: hace 5 min estaba a 36, ahora esta a 39.

Grafana = la pantalla del hospital
  - Muestra todos los termometros en una pantalla central.
  - Si alguna temperatura sube demasiado, se ve inmediatamente.
  - El medico (tu) puede mirar la pantalla y decidir que hacer.

Sin el termometro no hay datos.
Sin la pantalla no ves los datos.
Necesitas los dos.

7. Metricas que expone CampusEats

Nuestra app expone cuatro metricas propias con la libreria prom-client:

MetricaTipoQue mide
http_requests_totalCounterTotal de peticiones HTTP (con labels: method, endpoint, status_code)
http_request_duration_secondsHistogramCuanto tarda cada peticion (en segundos)
database_dishes_totalGaugeCuantos platos hay en la base de datos
app_uptime_secondsGaugeCuanto tiempo lleva la app corriendo

Ademas, prom-client expone automaticamente metricas del proceso Node.js: uso de memoria del heap, event loop lag, handles activos, etc.

|                  |       |                  |       |                  |
|   CampusEats    | ----> |   Prometheus     | ----> |    Grafana       |

|  /metrics        |       |                  |       |                  |
+------------------+       +------------------+       +------------------+
                                    ^
                                    |
                           +------------------+
                           |  kube-state      |
                           |  metrics         |
                           |  (datos de K8s)  |
                           +------------------+
  • CampusEats expone sus metricas en /metrics (peticiones, latencia, platos).
  • Prometheus recoge metricas tanto de CampusEats como de los componentes internos de Kubernetes (pods, nodos, deployments).
  • Grafana lo muestra todo en dashboards.
